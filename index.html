MySQL Core Concepts

Master the fundamental building blocks of database design

üóÑÔ∏è

Databases & Tables

A database is a container for related data. Within it, tables organize data into rows (records) and columns (fields). Think of a table like an Excel spreadsheet.

üîë

Primary Key

A unique identifier for each row in a table. No two rows can have the same primary key value. Example: user_id in a users table.

üîó

Foreign Key

Creates relationships between tables by referencing a primary key in another table. Ensures referential integrity across related data.

‚ö°

Indexes

Speed up data retrieval by creating a lookup structure. Like a book's index, they help MySQL find data quickly without scanning entire tables.

üìä

Data Types

Define what kind of data a column can store: INT (numbers), VARCHAR (text), DATE (dates), DECIMAL (precise numbers), etc.

‚úÖ

Constraints

Rules that enforce data integrity: NOT NULL (required), UNIQUE (no duplicates), DEFAULT (automatic values).

What is the difference between INNER JOIN and LEFT JOIN? Easy

‚ñº

INNER JOIN: Returns only the rows where there is a match in both tables. If a row in the left table doesn't have a matching row in the right table, it won't appear in the result.

LEFT JOIN (or LEFT OUTER JOIN): Returns all rows from the left table, and the matched rows from the right table. If there's no match, NULL values are returned for the right table's columns.

SQL Example 
-- INNER JOIN: Only users with orders
SELECT users.name, orders.total
FROM users
INNER JOIN orders ON users.id = orders.user_id;

-- LEFT JOIN: All users, even without orders
SELECT users.name, orders.total
FROM users
LEFT JOIN orders ON users.id = orders.user_id;

Explain ACID properties in MySQL Medium

‚ñº

ACID stands for Atomicity, Consistency, Isolation, and Durability - four key properties that guarantee reliable database transactions:

  • Atomicity: A transaction is treated as a single unit - either all operations succeed or all fail. If any part fails, the entire transaction is rolled back.
  • Consistency: A transaction brings the database from one valid state to another valid state, maintaining all defined rules and constraints.
  • Isolation: Concurrent transactions execute independently without interfering with each other. Changes made in one transaction are invisible to other transactions until committed.
  • Durability: Once a transaction is committed, the changes are permanent, even in case of system crashes or power failures.

Example: When transferring money between bank accounts, ACID ensures the debit and credit happen together (Atomicity), account balances remain valid (Consistency), other transactions don't see partial changes (Isolation), and the transfer persists (Durability).

What are indexes and when should you use them? Medium

‚ñº

An index is a data structure that improves the speed of data retrieval operations on a table. Think of it like a book's index - it helps you find information quickly without reading every page.

When to use indexes:

  • Columns frequently used in WHERE clauses
  • Columns used in JOIN conditions
  • Columns used in ORDER BY or GROUP BY
  • Columns with high cardinality (many unique values)

When NOT to use indexes:

  • Small tables (full table scan is faster)
  • Columns with frequent INSERT/UPDATE/DELETE operations (indexes slow these down)
  • Columns with low cardinality (few unique values like boolean)
SQL Example 
-- Creating an index
CREATE INDEX idx_email ON users(email);

-- Creating a composite index
CREATE INDEX idx_name_age ON users(last_name, first_name, age);

What is normalization? Explain 1NF, 2NF, and 3NF Hard

‚ñº

Normalization is the process of organizing data to reduce redundancy and improve data integrity. Here are the first three normal forms:

First Normal Form (1NF):

  • Each column contains atomic (indivisible) values
  • Each column contains values of a single type
  • Each row is unique
  • No repeating groups or arrays

Second Normal Form (2NF):

  • Must be in 1NF
  • All non-key columns are fully dependent on the primary key
  • No partial dependencies on composite keys

Third Normal Form (3NF):

  • Must be in 2NF
  • No transitive dependencies (non-key columns shouldn't depend on other non-key columns)
  • All columns depend only on the primary key

Example: A table storing (StudentID, StudentName, CourseID, CourseName, InstructorName) violates 2NF because CourseName and InstructorName depend on CourseID, not StudentID. Split into Students, Courses, and Enrollments tables to achieve 3NF.

Explain the difference between DELETE, TRUNCATE, and DROP Easy

‚ñº

DELETE:

  • Removes specific rows based on a WHERE condition
  • Can be rolled back (supports transactions)
  • Triggers are fired
  • Slower for large datasets (logs each row deletion)
  • Does not reset AUTO_INCREMENT counter

TRUNCATE:

  • Removes all rows from a table
  • Cannot be rolled back (DDL statement)
  • Triggers are NOT fired
  • Faster than DELETE for large tables
  • Resets AUTO_INCREMENT counter to initial value

DROP:

  • Removes the entire table structure and data
  • Cannot be rolled back
  • Frees up table space completely
  • Removes table definition from database
SQL Example 
DELETE FROM users WHERE age < 18;  -- Remove specific rows

TRUNCATE TABLE users;  -- Remove all rows, keep table

DROP TABLE users;  -- Remove table entirely

What are transactions and how do you use them? Medium

‚ñº

A transaction is a sequence of one or more SQL operations treated as a single unit of work. All operations must succeed together, or the database remains unchanged.

Key commands:

  • START TRANSACTION or BEGIN - Initiates a transaction
  • COMMIT - Saves all changes permanently
  • ROLLBACK - Undoes all changes since the transaction started
  • SAVEPOINT - Creates a point within a transaction to rollback to
SQL Example - Money Transfer 
START TRANSACTION;

-- Deduct from sender
UPDATE accounts 
SET balance = balance - 100 
WHERE account_id = 123;

-- Add to receiver
UPDATE accounts 
SET balance = balance + 100 
WHERE account_id = 456;

-- If both succeed, commit
COMMIT;

-- If any fails, rollback
ROLLBACK;

What is the difference between CHAR and VARCHAR? Easy

‚ñº

CHAR(n):

  • Fixed-length character string
  • Always uses exactly n bytes of storage (padded with spaces)
  • Faster for fixed-size data (e.g., country codes, phone numbers)
  • Maximum length: 255 characters

VARCHAR(n):

  • Variable-length character string
  • Uses only the space needed (plus 1-2 bytes for length)
  • Better for variable-size data (names, emails, addresses)
  • Maximum length: 65,535 characters
SQL Example 
CREATE TABLE users (
    country_code CHAR(2),        -- Always 2 chars: 'US', 'UK'
    name VARCHAR(100),          -- Variable: 'John' uses 4 bytes
    zip_code CHAR(5),           -- Fixed: '12345'
    email VARCHAR(255)         -- Variable length emails
);

Explain the difference between MyISAM and InnoDB storage engines Hard

‚ñº

InnoDB (Default in modern MySQL):

  • Transactions: Supports ACID transactions with COMMIT and ROLLBACK
  • Foreign Keys: Enforces referential integrity
  • Locking: Row-level locking (better for concurrent writes)
  • Crash Recovery: Automatic crash recovery
  • Performance: Better for applications with many INSERT/UPDATE operations
  • Use case: E-commerce, financial apps, any OLTP system

MyISAM:

  • Transactions: No transaction support
  • Foreign Keys: Does not support foreign keys
  • Locking: Table-level locking (slower for writes)
  • Speed: Faster for read-heavy workloads
  • Full-text search: Better full-text search capabilities (historically)
  • Use case: Read-heavy applications, data warehousing, archival data

2026 Recommendation: Use InnoDB for almost everything. MyISAM is largely deprecated and InnoDB now supports full-text search as well.

How do you optimize a slow MySQL query? Hard

‚ñº

Query Optimization Strategies:

  1. Use EXPLAIN: Analyze query execution plan 
    EXPLAIN SELECT * FROM users WHERE email = 'user@example.com';
  2. Add Indexes: Create indexes on columns used in WHERE, JOIN, ORDER BY
  3. Avoid SELECT *: Only select needed columns
  4. Use LIMIT: Reduce result set size when possible
  5. Optimize JOINs: Ensure JOIN columns are indexed
  6. Avoid Functions on Indexed Columns: 
    -- Bad: Function prevents index use
WHERE YEAR(created_at) = 2026

-- Good: Use index-friendly range
WHERE created_at >= '2026-01-01' AND created_at < '2027-01-01'
  7. Use Covering Indexes: Include all queried columns in the index
  8. Partition Large Tables: Split tables by date, region, etc.
  9. Cache Results: Use query cache or application-level caching (Redis, Memcached)
  10. Denormalize When Needed: For read-heavy workloads, some redundancy can improve performance

What are prepared statements and why should you use them? Medium

‚ñº

Prepared statements are pre-compiled SQL statements that separate SQL logic from data, offering two major benefits:

1. Security (SQL Injection Prevention):

  • User input is treated as data, not executable code
  • Prevents malicious SQL injection attacks

2. Performance:

  • SQL is parsed and optimized once, then reused with different parameters
  • Faster execution when running the same query multiple times with different values
PHP Example 
// Bad: Vulnerable to SQL injection
$sql = "SELECT * FROM users WHERE email = '" . $_POST['email'] . "'";

// Good: Protected with prepared statement
$stmt = $pdo->prepare("SELECT * FROM users WHERE email = ?");
$stmt->execute([$_POST['email']]);

// Or with named parameters
$stmt = $pdo->prepare("SELECT * FROM users WHERE email = :email");
$stmt->execute(['email' => $_POST['email']]);
